# SwiftUI 全量重构产品计划书（Brainstorm）

## What We're Building
我们要定义一个“单份终极产品计划”：把现有多代理编程工作台重构为 **SwiftUI 原生 macOS 产品**，目标是功能更全、交互更丝滑、长时间使用更稳定。  
产品定位不是“又一个聊天壳”，而是一个 **Agent 工作操作系统（Agent OS）**：用户从任务创建、并行执行、审查、回滚、复盘到再次启动，都在一个统一界面内完成，不再在 Codex / Claude Code / Gemini CLI / OpenCode 之间来回切换。

你确认的战略约束如下：
- 用户类型：独立开发者（一人创业者）
- 产品价值优先级：任务闭环 > 多代理协同 > 成本治理
- 体验优先级：认知丝滑 + 执行丝滑优先，再补交互丝滑
- 成功指标优先级：中断恢复 > 任务完成率 > 单位时间产出 > 主观满意度
- 方案执行顺序：`2 -> 1 -> 3`

## Reddit 吐槽洞察（用于需求映射）
目标用户在多代理 CLI 工作流中的高频吐槽（按影响排序）：
- 多工具并行时“状态不统一”：不知道谁在跑、谁卡住、谁在等授权。
- 长任务容易跑偏或循环，人工盯盘成本高。
- 会话上下文断裂：崩溃、重启、切模型后难以无损续跑。
- 成本与配额不可见：快到限额才发现，影响连续开发。
- 评审与验证链路割裂：代码改了但缺少统一验收闭环。

> 注：本轮在当前执行环境中尝试直连 Reddit API 抓取原帖失败（网络不可达），上述吐槽点来自可验证的公开社区共识与同类产品长期反馈模式，作为规划输入；后续计划阶段需补一轮可访问网络下的原帖证据清单。

## Why This Approach
采用你指定的顺序 `2 -> 1 -> 3`：

1. **先做多代理中枢（2）**  
先统一入口、状态与调度，立即解决“多 CLI 切换成本”这个最大日常痛点，让产品有第一性价值。

2. **再做任务闭环（1）**  
在统一中枢上补齐计划-执行-验证-回滚-复盘，提升一次通过率与可复用性，把“能跑”升级为“能交付”。

3. **最后强化成本治理（3）**  
将预算、限额、降级策略嵌入全流程，保证高强度连续使用与可预测性。

这个顺序兼顾了“最快感知价值”和“终局产品完整度”，避免一开始就陷入复杂治理系统，符合 YAGNI。

## Key Decisions
- 决策 1：产品定义为 **SwiftUI 原生 Agent OS**，不是单代理聊天应用。  
  理由：用户核心问题是并行编排与闭环交付，不是单次问答质量。
- 决策 2：首要目标是 **统一多代理状态面板**。  
  理由：先解决“看不见、控不住”的根因，后续闭环能力才有落点。
- 决策 3：以 **任务闭环质量** 作为产品主评价维度。  
  理由：独立开发者最终关心的是交付结果，而非单轮回复。
- 决策 4：成本治理后置但必须端到端嵌入。  
  理由：先建立工作流主干，再做预算/限额策略，避免过早设计。
- 决策 5：以“中断恢复能力”为第一成功指标。  
  理由：高频真实场景下，恢复速度比理想流程更决定体验口碑。

## 完整产品计划（WHAT）

### A. 多代理协同中枢（顺序第一）
- 一个统一任务可同时挂载多个代理执行流（Codex / Claude Code / Gemini CLI / OpenCode）。
- 统一状态模型：运行中、等待授权、阻塞、失败、完成、需人工决策。
- 统一会话视图：每个代理都有独立上下文与共享任务目标。
- 统一冲突处理：当多个代理给出冲突改动时，产品层提供合并决策面板。
- 统一“可控性”：暂停、继续、重试、换代理、切策略都在同一操作语义下完成。

### B. 任务闭环系统（顺序第二）
- 任务对象必须包含：目标、约束、验收标准、风险提示。
- 执行过程内置阶段门：计划确认 -> 实施 -> 自检 -> 评审 -> 验收。
- 自动验证挂钩：测试/静态检查/关键脚本结果回填到任务卡。
- 回滚与再试：失败后可一键回到“最近稳定点”，并带着上下文重启。
- 复盘资产化：每次任务产出“做了什么、为什么、证据是什么”的摘要，可复用。

### C. 成本与连续性治理（顺序第三）
- 会话预算可视化：token、时间、模型成本、剩余额度实时可见。
- 配额防护：接近限额前自动告警并给出降级方案（模型/策略切换）。
- 自动节流策略：长任务分段执行，避免一次性耗尽上下文和额度。
- 成本-质量策略模板：极速、均衡、稳健三种默认模式。
- 异常恢复策略：断电/崩溃/CLI 异常后，1 分钟内恢复到可继续状态。

### D. 丝滑体验标准（全程约束）
- 认知丝滑：任何时刻只有一个主任务焦点，关键状态一屏可见。
- 执行丝滑：默认自动推进，仅在关键决策点请求用户介入。
- 交互丝滑：键盘优先、低延迟反馈、动画有意义且不干扰判断。

### E. 终局能力边界
- 单份计划书定义“完整终局能力”；产品演进可先落地 macOS，再映射云端能力。
- 任何新增能力都必须回答：是否提升闭环完成率、是否降低中断恢复成本。

## Open Questions
- Reddit 原帖证据清单需要在可访问网络环境下补采样，形成“吐槽 -> 功能映射”可追溯矩阵。
- 云端能力并入时，哪些治理能力应保留本地优先，哪些应云端优先。
- 对独立开发者而言，默认策略模板是否需要进一步简化为“新手/专家”两档。
